# Concurrency, Transactions & Idempotency Audit

Scope: `ATOMIC_REQUESTS`, `select_for_update` (gapless `MatchEvent.sequence_no`), `transaction.on_commit` hooks (publish + advancement) and their ordering, idempotent writes via unique `event_id` (replay → 200), race conditions, and non-atomic multi-step writes that could corrupt state.

Method: every claim is cited to `file:symbol` + line range and verified against source (not the breadth-pass notes). No source was modified.

## Key architectural fact that frames everything below

`DATABASES["default"]["ATOMIC_REQUESTS"] = True` (`fixture/settings/base.py:110`) with **no** `non_atomic_requests` / per-view override anywhere in `apps/` or `fixture/` (grep confirmed; the only other hit is a test docstring). Consequences that recur in the findings:

1. Every DRF view body runs inside one request-level DB transaction. So every service-layer `with transaction.atomic()` is a **savepoint**, not an independent transaction.
2. Every `transaction.on_commit(...)` registered by a service fires **only at the outermost request commit**, in registration order — never at the inner `atomic()` block's exit. So "publish after commit" / "advance after commit" hooks for *all* mutations in one request fire together at the very end.
3. A `select_for_update()` row lock is held until the **request** ends, not until the inner block ends — lock duration is the whole request, widening contention windows.

| 1 | Walkover never advances the bracket (terminal-with-result fires advancement, but no score is set) | **High** | `apps/matches/services/state.py:33,67-69` (`_TERMINAL_WITH_RESULT`, `transition_match`); `apps/fixtures/services/advance.py:20-24`; `apps/matches/models.py:107-124` (`winner_id`/`loser_id`) | `transition_match(to=WALKOVER)` fires `_fire_advancement`, but it only sets `status`/`current_period` — it never sets `home_score`/`away_score`. `winner_id` returns `None` when `home_score is None or away_score is None`, so `advance_from_match` hits `if winner_id is None: return []` and does nothing. There is no other code path that sets a walkover score (grep over `home_score`/`away_score` shows only `record_score`, `recompute_score`, and the e2e demo). | The state machine must derive/record a walkover score (or define a walkover winner pointer) atomically with the transition so advancement can resolve. | 2 | Two independent writers of the cached `home_score`/`away_score` with no shared lock ordering → derived score and final score can diverge / clobber | **High** | `apps/matches/services/scoring.py:71-103` (`record_score`); `apps/matches/services/events.py:49-74,88-110` (`recompute_score` called from `record_match_event`) | The match score is written by two unrelated paths against the same columns: (a) `record_match_event` → `recompute_score` derives from the event log and does `Match.objects.filter(pk=...).update(home_score=..., away_score=...)`; (b) `record_score` sets `home_score`/`away_score` from the request body and flips status to COMPLETED. Both take `select_for_update` on the Match, so they serialize, but there is no invariant tying them together: a `record_score` of 2-1 followed by a late `record_match_event` goal recomputes to 3-1 (or back to 1-0 if events were never logged), silently overwriting the official result. The PRD says score is *derived* (invariant #4), yet `record_score` stores a non-derived value and `record_match_event` overwrites it. The two are reconciled by nothing. | Pick one source of truth. Either delete `record_score`'s score write and make the final score a pure projection of the event log, or forbid `record_match_event` once status is COMPLETED. The restructured engine should have exactly one writer for the cached score. |
| 3 | `record_match_event` idempotency pre-check is outside the lock and not backstopped by an IntegrityError catch → concurrent replay can 500 or create a duplicate side effect | **Medium** | `apps/matches/services/events.py:83-128`; unique constraint `MatchEvent.event_id` (`apps/matches/models.py:284`) | The replay guard `prior = MatchEvent.objects.filter(event_id=event_id).first(); if prior: return prior` runs *before* `transaction.atomic()` and is not re-checked under the lock. Two concurrent requests with the same `event_id` can both pass the pre-check, both enter the block, and the second `MatchEvent.objects.create(... event_id=...)` violates the unique constraint → `IntegrityError`. Because `ATOMIC_REQUESTS` is on, that IntegrityError marks the *whole request transaction* as broken and surfaces as a 500 (not the spec's "replay → 200"). Contrast `forms/services/responses.py:46-68`, which wraps the create in a savepoint and catches `IntegrityError` to return the existing row — the correct pattern that `events.py` lacks. The audit row (`emit_audit` with `idempotency_key=event_id`) would also be a second side effect attempted. | Wrap the create in a savepoint + `except IntegrityError: return MatchEvent.objects.get(event_id=event_id)` (mirror `submit_response`). Restructure idempotency into a single reusable helper so every verb gets the same race-safe replay-→-200 behavior instead of ad-hoc per-service pre-checks. |
| 4 | `record_score` idempotency keyed on AuditEvent, not on the mutated row, and pre-checked outside the lock | **Medium** | `apps/matches/services/scoring.py:64-69`; `apps/audit/services.py:45-48`; `AuditEvent.idempotency_key` unique (`apps/audit/models.py:48`) | Replay detection is `AuditEvent.objects.filter(idempotency_key=event_id, event_type="match_scored").first()` *before* the lock. Same race as #3: two concurrent identical `event_id` requests both miss, both lock the Match in turn, the first scores+writes the audit, the second now finds `status==COMPLETED` and raises `ValidationError("Cannot score a match in status 'completed'")` → 400, not the idempotent 200 the spec promises. (The unique `idempotency_key` also makes the second `emit_audit` create throw IntegrityError if it ever reached it.) The idempotency contract is on a *side-effect* table rather than the entity, so it is fragile. | Move idempotency to a first-class key on the entity/verb (or a dedicated idempotency table) checked under the same lock that performs the mutation; return the prior result on conflict. |
| 5 | Concurrent fixture generation can create duplicate brackets — `generate_single_elimination` has **no** idempotency guard and no lock | **Medium** | `apps/fixtures/services/generate.py:90-143` (`generate_single_elimination`); `apps/fixtures/views.py:30-43` (`GenerateFixturesView`) | `generate_round_robin` (line 42-46) and `generate_knockout_from_groups` (153-159) short-circuit if matches already exist, but `generate_single_elimination` does not — it computes `match_no = Match.objects.filter(...).count()` (line 103) and unconditionally `bulk_create`s a full bracket. | 6 | Advancement is non-atomic, unlocked, and does not cascade across rounds | **Medium** | `apps/fixtures/services/advance.py:16-46` (`advance_from_match`); fired from `state.py:73-80` / `scoring.py:99-102` on_commit | `advance_from_match` loops every match in the tournament, mutates dependents with `dep.save(update_fields=["home_team","away_team","updated_at"])`, and does **not** open a transaction, does **not** lock the dependents, and runs **after** the request commit (so the parent request's atomicity does not cover it). If it raises partway, some dependents are filled and others not, with no rollback — and `_fire_advancement` swallows the exception (`state.py:79-80`), so the partial state is silently left and never retried. It also does not recursively advance: filling a semifinal team does not itself trigger the next round (the next round only resolves when *that* match completes), which is correct, but the lack of a transaction means a multi-dependent fan-out (e.g. a loser_of + a winner_of both pointing at the same match) is not all-or-nothing. Two concurrent completions feeding the same dependent (home from match X, away from match Y) can interleave read-modify-write on the dependent row and lose one side. | Make advancement transactional and lock the dependents it edits (`select_for_update`), use targeted querysets (`home_source__match_id` lookups) instead of scanning all matches, and make it idempotent + retryable (it currently relies on at-most-once on_commit with no retry). |
| 7 | All `on_commit` hooks fire at the outermost request commit, so publish/advancement ordering is global to the request, not per-verb | **Low/Medium** | `events.py:127` (`publish_match_event`); `scoring.py:102` & `state.py:69` (`_fire_advancement`); `notifications/services/dispatch.py:34`; `forms/services/responses.py`; `ATOMIC_REQUESTS` `base.py:110` | Because `ATOMIC_REQUESTS=True`, an inner `transaction.atomic()` is a savepoint and `transaction.on_commit` defers to the *request* commit. A single request that records several events then transitions a match registers N `publish_match_event` + one `_fire_advancement`, all firing at the end in registration order. The intended guarantee "publish/advance strictly after the DB row is durable" holds, but: (a) the per-verb ordering the code comments imply (publish right after this event commits) is not what happens — everything batches; (b) if the request raises *after* a successful inner savepoint but before request end, the whole request rolls back and **no** hook fires (correct), but the caller may have already observed a 2xx-shaped object in memory; (c) hooks run synchronously in the request thread after commit, so a slow `group_send` (Redis) adds latency to the response and, on failure, only logs (best-effort) — delivery loss is silent. | Decide explicitly whether hooks are per-verb or per-request; if per-verb durability/ordering matters, the verbs need their own committed transactions (which conflicts with ATOMIC_REQUESTS) or an outbox table. The restructured live layer should use a transactional outbox + worker rather than in-request `on_commit` fan-out. |
| 8 | `recompute_score` does a full table scan + per-row Python aggregation under the match lock; O(events) work inside the critical section | **Low** | `apps/matches/services/events.py:49-74` | Every appended event re-derives the score by loading the voided-id set and iterating all `MatchEvent` rows for the match in Python, all while holding `select_for_update` on the Match (held until request end under ATOMIC_REQUESTS). For a long match this is repeated O(n) work per event under a row lock that also blocks `record_score`/transitions on the same match. Correctness is fine; contention/throughput is not. The VOID handling also only treats a row as voided if some VOID points at it (`voids__isnull=False`), and double-voiding the same event is not prevented (two VOIDs of the same target just both no-op the already-excluded event — benign but unaudited as a conflict). | Maintain the score incrementally (apply the single new event) or aggregate in SQL; keep the lock window minimal. Consider a unique partial constraint preventing two live VOIDs of the same target. |
| 9 | Gapless `sequence_no` is correctly serialized, but the guarantee depends entirely on the row lock + unique constraint; the Max+1 read is not by itself safe | **Info (correct, fragile)** | `events.py:88-93`; unique `(match, sequence_no)` (`apps/matches/models.py:294-298`) | `next_seq = Max(sequence_no)+1` is computed *after* `Match.objects.select_for_update().get(pk=...)`, so concurrent appenders serialize on the match row and cannot pick the same seq; the unique `(match, sequence_no)` constraint is the backstop. This is correct. The fragility: the safety lives in the discipline of always locking the match first — any future code path that appends a `MatchEvent` without taking the match lock (e.g. a bulk import, or advancement writing an event) would reintroduce a gap/duplicate race. There is no DB-level "next value" mechanism. | If sequence_no must stay gapless, encode it as a DB-side sequence per match or document the lock contract as load-bearing; the restructure should centralize event append behind one function so the lock can't be bypassed. |

## Cross-cutting summary for the restructuring

- **One idempotency mechanism.** Today there are at least two (entity `event_id` unique column vs. `AuditEvent.idempotency_key` lookup) and the pre-checks are uniformly **outside** the lock with no IntegrityError backstop except in `forms`. Centralize a race-safe "get-or-create on idempotency key under the lock, return prior on conflict → 200" helper and use it everywhere. This single change fixes findings #3, #4, #13, #14.
- **One writer for the cached score.** `record_score` (stores a typed-in value) and `recompute_score` (derives from events) both write `home_score`/`away_score` with no reconciliation (#2). Decide: derived-only, or stored-only; do not keep both.
- **Walkover must produce a winner** before/with firing advancement (#1) — the single highest-impact correctness bug found.
- **Advancement should be transactional, locked, targeted, and idempotent** (#6) and should fire from a transactional outbox rather than in-request `on_commit` (#7), because `ATOMIC_REQUESTS` makes `on_commit` a per-request batch with synchronous, best-effort, lossy delivery.
- **Generation must be lockable/idempotent** — `generate_single_elimination` is currently neither (#5).
- **Lock-then-read discipline** is load-bearing for gapless `sequence_no` (#9) and for not acting on stale team/score data (#10, #11); the restructure should make it impossible to append a `MatchEvent` or mutate a match without first taking the match lock.
